The_rule_of_hole

The rule of hole: If you fall into a hole, don't dig 对很多人来说，承认自己错了并不是一件容易办到的事情，即使他们看上去并不是一个固执的人。有时候是因为害羞，有时候是为了面子，有时候是为了某些利益，有时候是觉得自己能扭转局面，人会在意识到自己错了之后，继续沿着原路往前走，越陷越深。 关于如何成为一个职业的程序员，我们常常被教育要说到做到。能就是能，不能就是不能，不要说试试看。当然，我完全同意这些话，但是有些同学会把这句话理解为，如果我接了这个需求，我就一定要做到。 上面的两种说法看起来差不多，其实换了个角度，表达的意思就完全不同了。在评估需求的时候，我们当然要足够认真仔细，明白什么是你能做到的，什么是你做不到的，什么是你能做到但是让你觉得不对劲的。（如果你只把自己当做一个勤劳的码农，觉得评审需求是产品经理和架构师的事，那你并不是一个合格的工程师。如果你们公司不让你参加需求评审，那还是趁早换个地方吧。）但是当你在开发过程中，发现之前评审的需求中某个地方有逻辑漏洞，或者你发现你必须搞一堆很奇怪的东西（在代码里下毒）才能完成需求，你应该立即站出来，而不是闷着头做下去。 这是我在工作中学到最重要的一件事：掉坑别挖。

Copy_on_write

swift中的cow swift中有struct和class这两种数据结构，很多入门教程都会介绍说，struct和class最大的区别是，struct是值类型，而class是引用类型。换句话说，struct在传递的过程中是值传递，而class传的则是引用。 那么为什么需要这两种传值机制呢？从编码角度看，值传递似乎是一种更安全的选择。因为值传递在每次传递过程中，对象都会被拷贝一份，这样可以防止对象被意外修改。在多并发情况下，值传递更安全。但是在很多情况下，或许用户确实需要一个对象在不同的执行块共享，甚至共同修改，这时候引用传递显然更适合。更重要的是，引用传递是一种开销更小的传递方式，程序不需要为这次传递申请一块新的内存并拷贝对象，也不用考虑对象失去全部有效引用时回收这块新开辟的内存。 那有没有可能实现一种技术，让我们同时享受值传递和引用传递的优点呢？答案是”写时复制”(cow, copy-on-write)。顾名思义，cow就是说当使用值传递的时候，在新的代码块中依旧引用原来的对象，直到改变(写)对象的时候，才复制一份对象并引用新对象。在swift中，你可以使用mutating关键字来标识struct的某个方法是否会改变这个对象。当调用一个mutating方法时，才会真的复制一份对象。 cow与引用计数 英文好的同学建议直接读这篇instagram的原文 在instagram停用了python的垃圾回收。 linux内核有一个这样的机制：子进程启动时会共享父进程的内存页，但是只有当子进程试图去写这些内存页的时候，才会把这些内存拷贝到新的内存页，也就是我们说的”写时复制”。但是在python中，由于python使用引用计数来进行垃圾回收，这导致了每次读一个对象，都需要去写这个对象的引用数(加1)，所以所有的读就都变成了写，cow机制就失效了。为此，instagram关掉了python的gc…

kotlin中的可空类型

最近在公司使用Kotlin写后端服务（spring + mybatis），遇到optional的小问题，如下 fun getProject(projectCode: String): ProjectDO = projectMapper.getProjectByCode(projectCode) 上面这个方法根据项目号去数据库拿项目，从类型签名上看，getProject方法返回的是一个ProjectDO类型的对象，而且由于不是可空类型，所以不会是null。但是，如果数据库里面确实没有这个项目号的项目呢？试验一下 print(getProject("no such project")) // null 返回结果是null!! kotlin欺骗了我们，明明说好是非空，怎么就返回了一个null给我！ 其实这锅kotlin真不背。了解mybatis的同学应该知道，mybatis采用的是jdk代理的模式来代理Mapper接口。尽管我们在定义mapper接口时写明了返回非空类型的对象， projectMapper.getProjectByCode(projectCode): ProjectDO 但是在jvm上运行的时候，运行时类型可没有什么非空不非空的。查不到数据，mybatis的mapper的代理对象就返回了一个null给调用者，调用的地方拿这个对象赋值也好，返回也好，也不存在可空类型，拿到null就是null了。 所以kotlin的可空类型，只能在编译时帮你搞定一些程序员搞出来的空指针异常。而在一些使用了代理技术的地方，运行时还是会给你跑出一个null来，仍然需要你手动处理。 by the way，如果你阅读kotlin代码编译后的字节码，会发现其实kotlin在所有不是可空的变量上加上了@NotNull注解，其RetentionPolicy为class /** * Annotations are to be recorded in the class file by the compiler * but need not be retained by the VM at run time. This is the default * behavior. */ CLASS, 说白了，它可不会在运行时起作用。

我只是想要一个函数啊

从面向对象说起 显然，面向对象编程在很长一段时间内，都是最主流的一种编程范式。那到底什么是面向对象呢？为什么我们需要对象呢？面向对象的最重要特性有三个：封装，继承，多态。在我看来，多态无疑是这三者中最为重要的特性，它解决了软件架构中最为复杂的依赖问题，也就是我们常说的解耦。继承则解决了代码复用的问题，并实现了一定程度的抽象。而封装则使数据变得更安全，同时也隐藏了数据的存储结构与类型。 对象的本质 假设我们现在要写一个人员管理的软件，需要实现一个给人员年龄加 1 的功能。非面向对象的写法可能是 addOneYearAge(person) 我们拿到一个数据的引用，然后给其年龄加一。而面向对象的方式则可能是 person.growUpOneYear() 这里我们有一个对象，然后给对象发送了一个消息，通知它发生一些变化。 这两种写法都能实现我们的目标，那么那种更好呢？如果我们抛开运行性能，抛开运行时函数入口跳转等等奇怪的字眼，单纯的看看上面两段代码，我觉得他们是一样的（我相信肯定有人会说第二种更好，或许也有人会支持第一种）。但是，当你的数据有二十种改变的方式的时候，第一种写法会崩溃的更快，各种改变 person 的函数会被写在你永远想象不到的地方，而第二种写法则相对容易管理，因为你控制了数据的改变（如果你知道你在做什么的话）。 所以，对象是什么已经很显而易见了。对象是一堆数据的集合，以及可以操作这些数据的方法。 闭包 为什么要从面向对象扯到闭包？这两个东西八竿子打不着啊？ 从前我也是这么认为的，直到有一天我读了 ruby 作者松本行宏的书《代码的本质》。他说：对象和闭包是同一事物的两面。下面是一段简单的 js 中使用了闭包的代码。 function outside() { var a = 0; return function inside() { a++; console.log(a); }; } var f = outside(); f(); //1 f(); //2 f(); //3 我们可以看到 outside 函数返回了一个 inside 函数，inside 函数可以访问 outside 函数作用域内声明的变量 a。通常情况下，我们认为当函数返回时，其调用栈的栈帧(stack frame)以及其上的变量就会被销毁，所以当 outside 返回的时候，a 就应该不能访问了，但事实是 inside 函数可以始终访问到 a 变量。关于闭包可能导致的内存泄露问题不是本文讨论的重点。真正让我觉得有趣的依然是松本先生的那句话：对象和闭包是同一事物的两面。 其实，我们可以认为对象是通过类机制将过程（方法）封装到了数据里面，而闭包是通过作用域将数据封装到过程（方法）里面。它们的本质都是将数据与操作数据的过程（方法）放到一起，将数据与计算放在一起罢了。 如果你还是不能理解这段内容，请仔细阅读《sicp》的第二章。 日常黑 spring 下面开启吐槽时间，热爱 java 或者 spring 的同学请迅速撤离，或者准备好和我”撕逼”。

装逼的艺术

最近意外买到一本很有意思的书，叫做《假装的艺术》。这本书有个副标题，”一本让你看起来无所不知的书”。如今这个年代，无所不知几乎已经成为不可能的事情。 你听过 The fin 这个乐队吗？ 你知道去年最流行的编程语言是什么吗？ 你连这个游戏都没玩过吗？ 诺兰最新的那部电影你还没看？ 每当遇到这种问题，你可以直接了当地回答“不知道，没听过，没兴趣”。或者，你也可以摆出一副嘲笑，满不在乎的表情，bluffing 一下。因为我比较爱酒，所以就认真读了书里关于酒的一部分文字，和大家分享一下。 点什么酒最有派头 走进任意一间小酒吧，点一份简单的伏特加，或者加冰的波本威士忌，这样的日子已经一去不复返了。记住各种酒的牌子绝对是社交表演中的重要部分，就好比是仪式中一个必不可少的环节。天知道在什么场合，人们就会忽然展开关于名牌伏特加的大讨论。鉴于这种可能性，你可不能毫无准备。 伏特加闻起来会有点消毒水的味道，在聊到伏特加的时候，你首先要强调：伏特加尝起来都一样。这就避免了有人会说”smirnoff 牌伏特加在蒙眼品酒会里被选为最佳”这样无聊的内容。真正好的伏特加应该是完全没有味道的。苏联红牌伏特加(stolichnaya)虽然很不错，但是有点过时。芬兰伏特加(finlandia)和绝对伏特加(absolut)一度很流行，但基本也是我们父辈的酒了。时下最火的伏特加要属荷兰的坎特一号(Ketel One)。高级伏特加(luksusowa)来自波兰，原材料是土豆，这种酒让人醉得更彻底。 苏格兰威士忌要相对简单一点，因为不像伏特加那样会被掺兑得乱七八糟。现在最智慧的喝法，是在纯苏格兰威士忌里面加一点冰块，然后等它融化完。single malt 威士忌现在依然是公认的上选，产地最好是来自高地(highland)或者艾莱岛(Islay)。高地苏格兰威士忌要选 Macallan 纯麦威士忌。如果你喜欢烟熏味的话，可以选择拉加维林。 除了苏格兰威士忌，还有一种美国产的波本威士忌。关于波本酒，你需要知道的是：1，波本酒很烈。2.杰克丹尼不是真正的波本酒。 朗姆酒一般适合往酒里加一些水果或者果汁，比如柠檬和酸橙。不妨试试巴西产的卡萨撒(cachaca)或者朗姆酒兑的莫吉托(mojito)鸡尾酒。 喝龙舌兰(taquila)的窍门在于只喝最好的，最纯的那几种。喝过之后要记得补充水分。龙舌兰是墨西哥的国酒（类似于我们的茅台这类的吧），玛格丽特鸡尾酒的基酒就是 taquila。 不知不觉，就写了这么多字了。不懂酒的朋友现在一定觉得我是一个很懂酒的人了吧。其实，我啥也不懂，我只是在 bluffing 而已哈哈。 最后希望大家都能多一些真诚，少一些套路，做一个现充的人。在这样的时代，不知道真的没什么大不了的。

如何设计一个系统

描述使用场景 设计任何系统之前，首先需要搞清楚这个系统的功能是什么。这是任何系统设计中最重要的一步，你必须非常谨慎。你的系统需要完成什么功能？未来可能会加入什么功能？是否有隐藏需求的存在？你的用户是谁？你希望他们如何使用你的系统。这些问题都必须被严肃地回答。以设计一个twitter为例。 系统需要能够支持用户发送twitter。 需要能够支持用户follow（unfollow）其他用户。 用户可以在自己的timeline上面看到自己follow的用户发送的twitter。 用户是所有在可以访问我们服务的地区的会使用互联网的人。 希望用户每天能浏览三次以上时间轴，并在twitter上发现新的朋友，每天花费超过半小时时间在app上。 设定约束与假设 你的系统有多少用户？多少活跃用户?每天会创造多少的数据？数据的分布是怎样的？是否存在明显的热数据？数据的读写比率可能是多少？你需要多大的rps? 1000，000，000活跃用户 每天有100,000,000用户 每天会有100，000，000条新twitter 24小时内的twitter会被大量的读，7天后的twitter被阅读的频率会比较低。 数据的读写大约为20:1 ...... 根据这些约束和假设，你的数据库架构，缓存，服务器，网络等等一系列基础设施配置就都可以开始围绕它们来设计。 此外，明白什么能做到什么做不到是非常重要的，很多时候你需要做出一些取舍。CAP理论就是一个极好的例子：很多时候你需要放弃一定的可用性来保证一致性，有些时候你又可以放宽一致性的要求，来提升可用性。这些平衡很多时候不取决于你有多少开发资源，而是你在系统设计之初就必须做出的妥协。 高级设计 在大概了解系统需求以及基础设施的设计轮廓后，我们可以开始着手高级设计。我们需要画出系统主要的组件和连接。这一阶段的设计不需要很详细，但是要尽可能清晰和完整地画出系统的主要组成部分。在这一过程中，需要不断在脑中提醒自己前两个阶段得出的结论，确保不会做出与之相悖的设计。 我比较喜欢先画出几个关键的服务，然后画出数据库，再画出网关和缓存，当然也不要忘记trace,LB这些。可以用不同形状来表示不同类型的组件。 设计核心组件 对每一个核心组件进行详细深入的分析。这一个环节的设计关注细节，你需要设计各个服务的接口和数据模型，思考服务间的边界，尽可能的消除服务的隐藏需求和隐藏依赖。 度量设计 完成细节设计后，你需要重新review你的设计，确认和处理瓶颈以及一些限制，论述可能的解决办法和代价。在实际进入开发和使用后，我们必须时刻关注重要的metrics。机器的性能怎么样？请求的延时是多少？访问是否存在峰谷？每次迭代需要多少时间？有多少bug?bug的平均修复时长是多少？有多少bug的出现来自于最初设计的错误？所有这一切都需要被仔细审视，并在最早的时间发现问题。 Not using metrics in your development process is like trying to land a plane in a storm with your altimeter not working. 开发过程中如果没有metrics，就像是风暴中降落一台高度计坏了的飞机。
